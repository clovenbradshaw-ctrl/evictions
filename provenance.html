Davidson County Sessions Court Records
Primary Source
Data Collection Method
Eviction filing records (detainer warrants) are collected from public court filings in Davidson County General Sessions Court. Each record includes docket number, file date, case status, plaintiff (landlord), defendant (tenant), attorney information, property address, and geographic coordinates.

Xano API Data Feed

Event-Oriented (EO) Data Format
Data arrives in an event sourcing format where each case can have multiple temporal records showing changes over time. Each activity record includes:
Field Type Description
ts integer Unix timestamp in milliseconds when the event was recorded
ts_iso timestamp ISO 8601 formatted timestamp for human readability
op string Operation type (INS, ALT)
entity_id string Unique case identifier (docket number)
entity_type string Type of entity (e.g., 'eviction_case')
target JSON What is being operated on (entity ID and type)
context JSON Operation details (data for INS, changes for ALT)
frame JSON Interpretive context (source, actor, version)
// Event sourcing reduction: multiple events ‚Üí current state

  .map(group => group.sortBy('ts'))
  .replay() // Apply each operation to reconstruct state

Eviction Rate per 1,000 Renter Households
Calculated
How We Calculate This
To fairly compare eviction burden across districts with different renter populations, we normalize by calculating evictions per 1,000 renter households in each district.
eviction_rate_per_1k = (eviction_count / estimated_renters) √ó 1000
üìä
Eviction Count
Count of detainer warrant filings in each district from court records
üè†
Estimated Renters
Calculated as: total_occupied_units - owner_occupied_units (from ACS data)

Example: A district with 150 evictions and 4,730 renter households has a rate of (150 / 4,730) √ó 1000 = 31.7 per 1,000 renter households.

District Assignment Methodology
Geocoded
How Cases Are Assigned to Districts
Every property address obtained from CaseLink is geocoded via Google's Geocoding API to latitude/longitude coordinates, then we perform a point-in-polygon spatial join with Metro Council District boundaries (post-2021 redistricting per BL2021-1052, approved January 18, 2022).
‚ö†Ô∏è Limitations
‚Ä¢ Some addresses may fail to geocode (typos, new construction, PO boxes)
‚Ä¢ District boundaries updated in 2022; historical data uses current boundaries
‚Ä¢ Properties on district borders assigned to single district based on centroid

How We Identify Serial Filings
We group eviction filings by normalized property address and count the number of filings per address within a specified time period. Addresses with multiple filings suggest either serial eviction as a business practice or high-turnover properties.

Case Outcomes & Displacement Classification
Understanding what docket entries mean for actual displacement
‚ñº
‚ö†Ô∏è Critical Methodological Note
Court records track legal proceedings, not actual departures. Some tenants leave voluntarily after receiving eviction judgments to avoid sheriff removal. Others settle cases by agreeing to move in exchange for dismissal‚Äîdisplacement that appears in court records as a "win" for the tenant. We use a tiered classification system to indicate certainty levels for displacement.
Displacement Certainty Classification
Tiered System
6-Tier Classification System
We classify case outcomes by certainty of displacement, from definitive physical eviction to ambiguous dismissals.
TIER 1: Physically Evicted (Confirmed)
100% Certainty
What to Look For
DC SHERIFF PAPER RETURN WRIT or similar text indicating "executed writ"
What This Means
The sheriff physically removed the tenant and their belongings. This is the only outcome where we can confirm with 100% certainty that displacement occurred on a specific date.
WHERE docket_text LIKE '%SHERIFF%RETURN%WRIT%'
   OR docket_text LIKE '%EXECUTED%WRIT%'
   OR docket_text LIKE '%WRIT%EXECUTED%'
TIER 2: Eviction Authorized (Writ Issued)
80-90% Likely
What to Look For
WRIT OF RESTITUTION-D1 or WRIT OF POSSESSION without a subsequent sheriff return
What This Means
Judge authorized sheriff to remove tenant, but no record of sheriff actually executing it. Tenant likely moved voluntarily to avoid sheriff removal. However, writs can be cancelled, so just the existence of a writ doesn't guarantee displacement.
-- Cases with writ issued but no sheriff return
WHERE EXISTS (pleading LIKE '%WRIT%RESTITUTION%')
  AND NOT EXISTS (pleading LIKE '%SHERIFF%RETURN%')
TIER 3: Court-Ordered Eviction (Judgment, No Writ)
60-70% Likely
What to Look For
JUDGMENT-D1 POSSESSION without a writ being issued
What This Means
Landlord won at trial and judge ruled tenant must leave, but no writ was issued (yet). Tenant may have left voluntarily, or case may have settled post-judgment. Judgment creates legal obligation but doesn't confirm actual departure.
TIER 4: Default Judgment (Tenant Did Not Appear)
65-75% Likely
What to Look For
DEFAULT in docket entries or DEFAULT JUDGMENT
What This Means
Tenant didn't appear at hearing and landlord won by default. High correlation with eventual displacement, but some tenants settle before sheriff arrival.
TIER 5: Dismissed (Displacement Possible)
20-60% Likely
What to Look For
NON SUIT, DISMISS, DISMISSAL
‚ö†Ô∏è This is Tricky
Dismissal can mean either outcome:

1. Tenant stayed: Landlord dropped case, tenant paid arrears, or settlement allowed tenant to remain.

2. Tenant agreed to move: In some cases, the tenant agrees to move in exchange for dismissing the eviction. This shows up in CaseLink as a non-suit (plaintiff requests dismissal) even though the renter had to move.
Dismissal Type Pattern
With settlement language DISMISS + SETTLEMENT/AGREEMENT
Voluntary non-suit NON SUIT, PLAINTIFF DISMISS
General dismissal DISMISS (no context)
TIER 6: Judgment for Tenant
5-10% Likely
What to Look For
JUDGMENT FOR DEFENDANT or similar language
What This Means
Tenant won the case‚Äîlandlord's eviction claim was rejected. Tenant likely stayed, though some may have moved anyway for other reasons.
Recommended Reporting Language
Best Practice
How We Report Outcomes
Instead of saying "X people were evicted," we use precise language that reflects our certainty level:

Example reporting format:

"Of [X] eviction filings in 2024:
‚Ä¢ [Y] cases resulted in sheriff-executed physical evictions (confirmed displacement)
‚Ä¢ [Z] cases had writs authorizing eviction but no sheriff execution record (likely displacement through voluntary departure)
‚Ä¢ [A] cases ended in default judgments (high likelihood of displacement)
‚Ä¢ [B] cases were dismissed, including settlements where tenants may have agreed to move in exchange for dismissal"

2. The "Renter Households" Calculation (The Key Denominator)
Calculated
Critical Normalization Figure
This is the critical figure used to normalize eviction counts. Here's exactly how it's calculated from district-data.json metadata:
Step Source Description
Step 1 Census Table B25002 Get total_housing_units
Step 2 Census Table B25002 Get pct_occupied
Step 3 Census Table B25075 Get owner_occupied_units
Step 4 Calculated total_occupied = total_housing_units √ó (pct_occupied / 100) (rounded)
Step 5 Calculated estimated_renters = total_occupied - owner_occupied_units
Concrete Example - District 2
total_housing_units = 8,957
pct_occupied = 91.8%
owner_occupied_units = 3,493

total_occupied = 8,957 √ó 0.918 = 8,223 (rounded)
estimated_renters = 8,223 - 3,493 = 4,730 renter households
Validation Note
Davidson County total of 146,473 closely matches 146,398 "Occupied units paying rent" from Census Table B25070 (page 56). The calculated estimated_renters is validated against Census reported "occupied units paying rent"‚Äîthey match within 0.05%.
3. The "Per 1,000 Renters" Calculation
Calculated
Core Formula
Output Unit: "evictions per 1,000 renter households"
evictionRatePer1k = (eviction_count / estimated_renters) √ó 1000
Why √ó 1000?
The raw calculation eviction_count / estimated_renters produces a small decimal (e.g., 0.0317). Multiplying by 1000 converts this to a rate "per thousand" that's easier to interpret and compare.
Concrete Example - District 2
eviction_count = 150 (hypothetical)
estimated_renters = 4,730

Raw rate: 150 / 4,730 = 0.0317
Per 1,000: 0.0317 √ó 1000 = 31.7 evictions per 1,000 renter households

Interpretation: For every 1,000 renter households in District 2, approximately 31.7 faced an eviction filing.
Rationale
Normalizes for varying renter population sizes across districts, enabling fair comparison of eviction burden.
4. Annualization (When Data Covers Less Than a Full Year)
Calculated
Partial Year Data Handling
When the eviction data doesn't span a full year, the code annualizes the rate for consistent comparison.
// Calculate per-month average
const evictionsPerMonth = monthsOfData > 0 ? evictionCount / monthsOfData : 0;

// Annualize to yearly
const annualizedCount = evictionsPerMonth * 12;

// Calculate eviction rate per 1,000 renter households
const evictionRate = estimatedRenters > 0 ?
    ((annualizedCount / estimatedRenters) * 1000) : 0;
Full Formula with Annualization
evictionRatePer1k = (eviction_count √ó (365 / days_of_data) / estimated_renters) √ó 1000

Or equivalently:
evictionRatePer1k = ((monthly_avg √ó 12) / estimated_renters) √ó 1000

Correlation Coefficient Calculation (Pearson's r)
Standard Pearson correlation coefficient formula used to measure linear relationship strength.
function calculateCorrelation(xValues, yValues) {
  const n = xValues.length;
  const sumX = xValues.reduce((a, b) => a + b, 0);
  const sumY = yValues.reduce((a, b) => a + b, 0);
  const sumXY = xValues.reduce((total, x, i) => total + x * yValues[i], 0);
  const sumX2 = xValues.reduce((total, x) => total + x * x, 0);
  const sumY2 = yValues.reduce((total, y) => total + y * y, 0);

  const numerator = n * sumXY - sumX * sumY;
  const denominator = Math.sqrt(
    (n * sumX2 - sumX * sumX) * (n * sumY2 - sumY * sumY)
  );

  return numerator / denominator; // Pearson's r
}

Event-Oriented (EO) Architecture
System Design
Event Sourcing Pattern
Data is retrieved from a backend API that implements event sourcing. Court records are scraped multiple times, and changes detected between scrapes are stored as events. Rather than overwriting data, each change is recorded as a new event, providing a complete history of how case information evolved over time.
Component Description
Event Types (op) INS (insert), ALT (alter)
Temporal Key ts (Unix timestamp in ms) + ts_iso (ISO 8601)
Entity Key entity_id (unique case identifier / docket number)
Target JSON object specifying what is being operated on (entity ID and type)
Context JSON object with operation details (data for INS, changes for ALT)
Frame JSON object with interpretive context (source, actor, version)
// Event sourcing reduction: multiple events ‚Üí current state
events.groupBy('entity_id')
  .map(group => group.sortBy('ts'))
  .replay() // Apply each operation to reconstruct state
Data Freshness & Limitations
System Info
Update Frequency
Court records are retrieved periodically from the public data source. There may be a lag between when a case is filed and when it appears in our data. Case outcomes may update as proceedings progress.
