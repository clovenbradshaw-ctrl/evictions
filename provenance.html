Davidson County Sessions Court Records
Primary Source
Data Collection Method
Eviction case records are collected from public court filings in Davidson County General Sessions Court. Each record includes docket number, file date, case status, plaintiff (landlord), defendant (tenant), attorney information, property address, and geographic coordinates.

Event-Oriented (EO) Data Format
Data arrives in an event sourcing format where each case can have multiple temporal records showing changes over time. Each activity record includes:
Field Type Description
logged_at timestamp When the change was recorded (ISO format)
action string Type of change (e.g., 'new_case', 'address_update')
docket_number string Unique case identifier (primary key for deduplication)
case_data object Nested case information (parties, dates, pleadings, judgments)
// Event sourcing reduction: multiple records ‚Üí current state
records.groupBy('docket_number')
  .map(group => group.sortBy('logged_at').last())
  .mergeFields() // Concatenate arrays like pleadings, judgments

Eviction Rate per 1,000 Renter Households
Calculated
How We Calculate This
To fairly compare eviction burden across districts with different renter populations, we normalize by calculating evictions per 1,000 renter households in each district.
eviction_rate_per_1k = (eviction_count / estimated_renters) √ó 1000
üìä
Eviction Count
Count of detainer warrant filings in each district from court records
üè†
Estimated Renters
Calculated as: total_occupied_units - owner_occupied_units (from ACS data)

Example: A district with 150 evictions and 4,730 renter households has a rate of (150 / 4,730) √ó 1000 = 31.7 per 1,000 renter households.

District Assignment Methodology
Geocoded
How Cases Are Assigned to Districts
Each eviction filing includes a property address. We geocode addresses to latitude/longitude coordinates, then perform a point-in-polygon spatial join with Metro Council District boundaries (post-2021 redistricting per BL2021-1052, approved January 18, 2022).
‚ö†Ô∏è Limitations
‚Ä¢ Some addresses may fail to geocode (typos, new construction, PO boxes)
‚Ä¢ District boundaries updated in 2022; historical data uses current boundaries
‚Ä¢ Properties on district borders assigned to single district based on centroid

Case Outcomes & Displacement Classification
Understanding what docket entries mean for actual displacement
‚ñº
‚ö†Ô∏è Critical Methodological Note
Court records track legal proceedings, not actual departures. Some tenants leave voluntarily after receiving eviction judgments to avoid sheriff removal. Others settle cases by agreeing to move in exchange for dismissal‚Äîdisplacement that appears in court records as a "win" for the tenant. We use a tiered classification system to indicate certainty levels for displacement.
Displacement Certainty Classification
Tiered System
6-Tier Classification System
We classify case outcomes by certainty of displacement, from definitive physical eviction to ambiguous dismissals. This system was developed with input from eviction defense attorneys.
TIER 1: Physically Evicted (Confirmed)
100% Certainty
What to Look For
DC SHERIFF PAPER RETURN WRIT or similar text indicating "executed writ"
What This Means
The sheriff physically removed the tenant and their belongings. This is the only outcome where we can confirm with 100% certainty that displacement occurred on a specific date.
WHERE docket_text LIKE '%SHERIFF%RETURN%WRIT%'
   OR docket_text LIKE '%EXECUTED%WRIT%'
   OR docket_text LIKE '%WRIT%EXECUTED%'
TIER 2: Eviction Authorized (Writ Issued)
80-90% Likely
What to Look For
WRIT OF RESTITUTION-D1 or WRIT OF POSSESSION without a subsequent sheriff return
What This Means
Judge authorized sheriff to remove tenant, but no record of sheriff actually executing it. Tenant likely moved voluntarily to avoid sheriff removal. However, writs can be cancelled, so just the existence of a writ doesn't guarantee displacement.
-- Cases with writ issued but no sheriff return
WHERE EXISTS (pleading LIKE '%WRIT%RESTITUTION%')
  AND NOT EXISTS (pleading LIKE '%SHERIFF%RETURN%')
TIER 3: Court-Ordered Eviction (Judgment, No Writ)
60-70% Likely
What to Look For
JUDGMENT-D1 POSSESSION without a writ being issued
What This Means
Landlord won at trial and judge ruled tenant must leave, but no writ was issued (yet). Tenant may have left voluntarily, or case may have settled post-judgment. Judgment creates legal obligation but doesn't confirm actual departure.
TIER 4: Default Judgment (Tenant Did Not Appear)
65-75% Likely
What to Look For
DEFAULT in docket entries or DEFAULT JUDGMENT
What This Means
Tenant didn't appear at hearing and landlord won by default. High correlation with eventual displacement, but some tenants settle before sheriff arrival. National data shows 84% of tenants don't appear at hearings.
TIER 5: Dismissed (Displacement Possible)
20-60% Likely
What to Look For
NON SUIT, DISMISS, DISMISSAL
‚ö†Ô∏è This is Tricky
Dismissal can mean either outcome:

1. Tenant stayed: Landlord dropped case, tenant paid arrears, or settlement allowed tenant to remain.

2. Tenant agreed to move: Per eviction defense attorneys: "We often represent renters in conduct evictions... Often the tenant will agree to move in exchange for dismissing the eviction. That shows up in CaseLink as a non-suit (plaintiff requests dismissal) even though the renter had to move."
Dismissal Type Pattern Est. Displacement
With settlement language DISMISS + SETTLEMENT/AGREEMENT 50-60%
Voluntary non-suit NON SUIT, PLAINTIFF DISMISS 40-50%
General dismissal DISMISS (no context) 20-30%
TIER 6: Judgment for Tenant
5-10% Likely
What to Look For
JUDGMENT FOR DEFENDANT or similar language
What This Means
Tenant won the case‚Äîlandlord's eviction claim was rejected. Tenant likely stayed, though some may have moved anyway for other reasons. This is rare (Memphis: 2% tenant win rate).

Technical Methodology
Event sourcing, data processing, and validation
‚ñº
Event-Oriented (EO) Architecture
System Design
Event Sourcing Pattern
Data is retrieved from a Xano backend API that implements event sourcing. Rather than storing just the current state, the system stores a log of all changes (events) that have occurred to each case. This provides a complete audit trail and enables reconstruction of case state at any point in time.
Component Description
API Endpoint xvkq-pq7i-idtl.n7d.xano.io/api:gx-QnxD1/detainer_case
Event Types new_case, address_update, status_change, pleading_added
Temporal Key logged_at (ISO 8601 timestamp)
Entity Key docket_number (unique case identifier)
Activity Stream Reduction
Calculated
Reducing Events to Current State
Each case can have multiple event records showing changes over time. We reduce these to the current state by grouping by docket_number, sorting by logged_at, and taking the most recent values while intelligently merging array fields.
// Step 1: Group events by case
events.groupBy('docket_number')

// Step 2: Sort each group by timestamp
  .map(group => group.sortBy('logged_at'))

// Step 3: Reduce to current state
  .map(group => {
    // Take latest scalar values
    // Concatenate + dedupe array fields (pleadings, judgments)
    // Deep merge nested case_data objects
  })
Field Type Merge Strategy
Scalar (status, address) Take most recent non-null value
Arrays (pleadings, judgments) Concatenate and deduplicate by ID
Nested objects (case_data) Deep merge with newer values overwriting
Timestamps Preserve _logged_at for audit trail
Incremental Sync
Performance
Smart Data Fetching
Instead of fetching all records on each load, we use the logged_at.gt query parameter to fetch only records updated since the last sync. This reduces API calls and improves load times.
// First load: fetch all records
GET /detainer_case?per_page=100

// Subsequent loads: fetch only updates
GET /detainer_case?logged_at.gt=2026-01-15T10:30:00Z
Field Normalization
Calculated
Standardizing Data
Court records may have inconsistent field formats (PascalCase vs. snake_case, varying date formats, address abbreviations). We normalize these for consistent analysis.
Field Type Normalization Applied
Dates Parsed to ISO 8601 format (YYYY-MM-DD)
Addresses Standardized abbreviations (ST‚ÜíStreet, AVE‚ÜíAvenue)
Names Trimmed whitespace, standardized case
Districts Geocoded from coordinates to district number
Renter Household Estimation
Calculated
How We Calculate Renter Counts
The Census doesn't directly report renter households by council district. We calculate this from available ACS data on housing occupancy and ownership.
total_occupied = total_housing_units √ó (percent_occupied / 100)
estimated_renters = total_occupied - owner_occupied_units
pct_renters = (estimated_renters / total_occupied) √ó 100

Validation: Our county-level estimate of 146,473 renter households closely matches the Census Bureau's reported 146,940 renter households from ACS Table B25070.

Data Freshness
System Info
Update Frequency
Court records are retrieved periodically from the public data source. There may be a lag between when a case is filed and when it appears in our data. Case outcomes may update as proceedings progress.
